# Production Readiness Plan - Hades Bug Root Cause Awareness - 2026-07-07

## Executive Summary

Questo piano copre il restante lavoro per arrivare all'ambizione: un agente
locale che puo' spiegare la causa precisa di un bug anche quando, durante la
diagnosi, non ha accesso diretto al filesystem della codebase.

La premessa tecnica va resa esplicita: "senza accesso al source code" non puo'
significare "senza nessuna evidenza di codice mai raccolta". Una root cause
precisa richiede almeno una di queste fonti pregresse e verificabili:

- indice strutturale aggiornato del progetto;
- grafo codice/framework aggiornato;
- evidence pack del bug, con stack trace, log, test falliti, request context e
  versione deploy;
- source slices bounded e policy-compliant per i punti chiamati in causa;
- storia risolta di bug simili e fix gia' verificati.

Stima di prontezza dopo le slice gia' completate: circa 40%.

Il 40% gia' ottenuto include memoria backend condivisa, search live
query-aware, raw chunks esclusi per default, artifact `hades.git_tree.v1` con
`project_index`, timeout live search aggressivo, e prefetch memoria reiniettato
in ogni chiamata LLM del tool loop.

Il restante 60% non e' un problema di "piu' memoria". E' un problema di
contratto causale:

1. acquisire evidenza di bug in modo tipizzato;
2. indicizzare davvero codice PHP/Laravel, dipendenze, DB, route, job, eventi e
   punti di ingresso;
3. rendere recuperabili source slices e graph facts con ACL, retention,
   redaction e freshness;
4. insegnare all'agent un workflow di root-cause investigation che blocca claim
   esatti quando evidenza o indice sono stale;
5. provare tutto con test no-codebase, dove il filesystem locale viene
   deliberatamente rimosso.

Il piano e' cross-layer:

- backend: sistema autorevole di memoria, evidence, graph, source slices e
  freshness;
- agent: orchestratore, tool service-gated, policy di confidenza e risposta;
- frontend/desktop/dashboard: intake, stato copertura, review policy e lettura
  evidence;
- operations: test, metriche, runbook e retention.

TencentDB Agent Memory resta escluso: l'architettura attuale non ha bisogno di
un nuovo memory store vendor-specific. Serve invece completare il contratto
Hades gia' avviato.

## User Answers And Assumptions

Risposte e vincoli confermati:

- TencentDB Agent Memory e' escluso.
- L'obiettivo e' login da qualunque device mantenendo memoria.
- La memoria progetto deve essere backend-owned e condivisa tra sessioni/device.
- L'agent deve essere project-aware in modo evidence-backed, non approssimato.
- Il caso target piu' difficile e' spiegare la causa precisa di un bug senza
  accesso live al source code.

Fatti tecnici confermati:

- Il prefetch memoria esterna viene calcolato a inizio turno e riusato in ogni
  API request del tool loop.
- Hades sa cercare memoria/wiki/artifacts via backend live search; il graph
  search agent usa artifact search live sotto il dominio `artifacts`.
- L'indice progetto attuale ha metadata-first tree/index, una prima graph
  extraction PHP/Laravel, graph search agent su artifact live e source slices
  bounded/redatte; non contiene ancora AST PHP/Laravel completo.
- `populate_backend_ast` produce `hades.php_graph.v1` per PHP/Laravel e
  `hades.symbols.v1` per Python.
- Il backend ha bug evidence, awareness/freshness status, source slice store e
  diagnosis reports; restano da rafforzare stale-blocking automatico e suite
  no-codebase.

Assunzioni operative:

- "Causa precisa" significa una risposta che cita evidenze verificabili e puo'
  dire "non determinabile" quando mancano dati.
- "Senza accesso al source code" significa senza filesystem locale nel momento
  della diagnosi, non senza avere mai indicizzato/salvato evidenza source-derived.
- Source slices possono essere salvate solo se policy progetto/utente lo
  permette; altrimenti il backend deve sapere chiedere letture bounded locali.
- Il toolset core deve restare stretto: nuove capacita' Hades devono essere
  service-gated provider tools, CLI+skill o API backend, non core tools.
- Backend e local repo possono avanzare in parallelo con branch coordinati.

Decisioni aperte per il proprietario:

- Policy source: metadata-only, selected source slices, full evidence pack, o
  per-project policy?
- Retention: quanti giorni per logs, source slices, stack trace, request
  payload, diagnosis report e resolved bug memory?
- Confidenza: quando un agent puo' dire "causa precisa" invece di "ipotesi
  probabile"?
- Lingue/framework prioritari dopo Laravel/PHP e Python.
- Modello identita': memoria personale cross-device separata dalla memoria
  progetto, o unificazione con ACL forti?

## P0 Blocking Foundations

### P0-1 No-Codebase Diagnosis Contract

- Area: Backend, agent, docs.
- Objective: Definire cosa l'agent puo' affermare quando non vede il filesystem
  locale e quali evidenze servono per una root cause precisa.
- Evidence: Il backend remoto ha una policy simile in
  `ServerAgentWorkService.php`: rispondere solo dal contesto fornito e dire cosa
  manca. Il provider locale oggi invita a cercare exact evidence ma non impone
  una scala di confidenza.
- Affected paths:
  - Local: `docs/hades/backend.md`, `docs/hades/operations.md`,
    `docs/hades/openapi-hades-v1.json`, `docs/backend-agent-coordination.md`,
    eventuale nuova skill `skills/hades-bug-diagnosis`.
  - Remote: docs API backend e seed/capability catalog.
- What/How:
  - Introdurre livelli di evidenza: `metadata_only`, `graph_supported`,
    `source_slice_supported`, `runtime_evidence_supported`,
    `verified_root_cause`.
  - Introdurre livelli di risposta: `not_enough_evidence`, `hypothesis`,
    `likely_cause`, `precise_cause`.
  - Definire che `precise_cause` richiede almeno: bug evidence, graph path,
    freshness valida e source slice o fact verified-from-code.
  - Definire formato di citazione: ogni claim causale deve puntare a evidence
    id, artifact id, source slice id, log id o resolved bug id.
  - Definire comportamento stale: niente claim precisi se HEAD o coverage non
    sono compatibili.
- Dependencies: Nessuna dipendenza tecnica, ma serve decisione policy source.
- Acceptance criteria:
  - Documentazione contiene definizione di "no-codebase mode".
  - OpenAPI o docs Hades elencano campi `confidence`, `evidence_refs`,
    `freshness`, `coverage`.
  - Le risposte agent in diagnosis mode hanno disclaimer solo quando serve, non
    come rumore fisso.
- Verification:
  - Test unit su formatter/policy agent per impedire `precise_cause` senza
    evidence refs.
  - Test docs/OpenAPI se presente.
  - Review manuale di 3 esempi: metadata-only, likely cause, precise cause.

  tipizzato, non come note generiche.
- Evidence: Backend oggi ha `project_memory_entries` generico e `hades_agent_jobs`
  capability-based, ma nessun endpoint o tabella per bug case/evidence.
- Affected paths:
  - Remote: nuove migrations per `hades_bug_reports`,
    `hades_bug_evidence_items`, `hades_bug_diagnoses`; nuovi controller
    `BugReportController`, `BugEvidenceController`, routes
    `/api/hades/v1/bug-reports`, `/api/hades/v1/bug-evidence`.
  - Local: `hermes_cli/hades_backend_client.py`,
    `hermes_cli/hades_backend_cmd.py`, `hermes_cli/hades_backend_db.py`,
    `plugins/memory/hades_backend/__init__.py`.
  - Tests: remote `tests/Feature/Hades`, local `tests/hermes_cli`,
    `tests/agent`.
- What/How:
  - Definire `bug_report`: project, workspace, title, symptom, severity,
    environment, affected route/task/session, observed_at, reporter.
  - Definire `evidence_item`: kind (`stack_trace`, `log_excerpt`,
    `failing_test`, `http_request`, `http_response`, `browser_console`,
    `deploy_version`, `config_snapshot`, `user_steps`, `screenshot_ref`),
    payload bounded, redactions, sha256, source, occurred_at.
  - Aggiungere local outbox per salvare evidence quando backend offline.
  - Aggiungere CLI `hades backend bug submit` o comando equivalente.
  - Consentire all'agent di creare evidence solo con payload redatto e bounded.
  - Collegare evidence a memory/wiki/artifact refs, non duplicare raw content in
    `project_memory_entries`.
- Dependencies: P0-8 redaction/retention minima; schema API backend.
- Acceptance criteria:
  - Un bug case puo' essere creato con stack trace e failing test senza source
    live.
  - Ogni evidence item ha hash, kind, size, redaction count e retention class.
  - Backend rifiuta evidence troppo grande, segreta o senza project binding.
  - Local sync ritenta upload outbox in modo idempotente.
- Verification:
  - Remote feature tests su create/list/search bug evidence.
  - Local client tests con HTTP mock.
  - Redaction tests per token/API key nei payload.
  - E2E con backend non raggiungibile -> outbox -> sync riuscito.

### P0-3 Freshness And Coverage Gate

- Area: Backend, local sync, agent.
- Objective: Impedire root cause precise quando indice, source slices o bug
  evidence non sono coerenti col commit/deploy che ha prodotto il bug.
- Evidence: `MemorySearchController` restituisce `workspace_head_commit` e
  `index_status=live_query`, ma non confronta artifact indexed commit,
  coverage o stale reason. Local `sync_git_tree` non include ancora un gate
  consumabile dal diagnosis workflow.
- Affected paths:
  - Remote: `MemorySearchController`, artifact storage/search, workspace binding
    model, nuove query coverage.
  - Local: `hermes_cli/hades_backend_jobs.py`,
    `hermes_cli/hades_backend_status.py`, `plugins/memory/hades_backend`.
  - UI: `web/src/pages/BackendPage.tsx`,
    `apps/desktop/src/app/chat/composer/status-stack/hades-backend-status.ts`.
- What/How:
  - Salvare per artifact: `indexed_head_commit`, `indexed_at`,
    `workspace_head_commit`, `dirty_state`, `coverage_summary`.
  - Aggiungere endpoint/status `project-awareness/status` o estendere
    capabilities/status con coverage per domains: routes, graph, source slices,
    bug evidence, tests, logs.
  - Calcolare `freshness_status`: `current`, `stale`, `unknown`,
    `partial`, `policy_blocked`.
  - Il tool agent deve ricevere freshness in ogni search/fetch response.
  - Il workflow diagnosis deve bloccare `precise_cause` se bug deploy commit
    non e' coperto.
- Dependencies: P0-1 contract; P0-4/P0-5 for coverage domains completi.
- Acceptance criteria:
  - Status mostra se il progetto e' diagnosticabile senza filesystem.
  - Search results distinguono "matching ma stale" da "matching current".
  - L'agent non produce claim precisi quando freshness e' stale/unknown.
  - Manual sync aggiorna coverage e rimuove stale warning.
- Verification:
  - Tests unit su stale comparison commit.
  - Feature tests remote su freshness response.
  - Agent tests che simulano stale search e verificano risposta conservative.
  - UI tests per warning coverage.

- Evidence: prima di questa slice `populate_backend_ast` leggeva solo `.py`;
  `project_index` vedeva route, dependencies e migrations ma non controller
  methods, service calls, model relationships, events, jobs, policies,
  middleware o DB usage.
- Affected paths:
  - Local: `hermes_cli/hades_backend_jobs.py`, possibili nuovi moduli
    `hermes_cli/hades_php_indexer.py`, `hermes_cli/hades_laravel_indexer.py`.
  - Remote: `ArtifactController`, `MemorySearchController`, OpenAPI,

  - Tests: `tests/hermes_cli/test_hades_backend_sync_runner.py`, remote
    `tests/Feature/Hades`.
- What/How:
  - Aggiungere schema `hades.php_graph.v1` o `hades.code_graph.v1`.
  - Estrarre:
    - routes -> controller/action/middleware/name/file/line;
    - controllers -> methods, request validation, service calls;
    - services -> public methods, injected deps, called repos/clients;
    - models -> table, fillable/casts/relations/scopes;
    - migrations -> tables, columns, indexes, foreign keys;
    - jobs/events/listeners/commands/schedulers;

- Acceptance criteria:
  - Un route/controller/service bug puo' essere collegato a symbols e call edges.

  - Backend memory search o graph search trova symbol, route e edge.
  - Il graph non include secret/raw file content.
- Verification:
  - Fixture Laravel mini-app con route, controller, service, model, migration.
  - Local tests su graph extraction deterministica.
  - Remote feature tests su upload/search `hades.php_graph.v1`.

## Risks And Decisions

Rischi principali:

- Falsa precisione: l'agent potrebbe parlare con sicurezza quando ha solo
  metadata. Mitigazione: P0-1 e P0-3 come gate hard.
- Privacy/source leakage: source slices e logs possono contenere segreti.
  Mitigazione: P0-8 prima di qualunque store content-bearing.
- Search rumorosa: LIKE/artifact JSON non basta per causal debugging.
  Mitigazione: P1-3 search strutturata e graph traversal.
- Staleness invisibile: un indice vecchio puo' produrre cause sbagliate.
  Mitigazione: freshness coverage in ogni response e UI.
- Coupling core: aggiungere tools core romperebbe la filosofia Hermes.
  Mitigazione: provider tools service-gated e backend APIs.
- Over-indexing costoso: indicizzare tutto puo' essere lento/caro.
  Mitigazione: source slices selected, incremental indexing, retention.

Decisioni da prendere prima di implementare source slices:

- policy default per progetti privati: off, selected-on-approval, o on;
- retention default di source slices e runtime logs;
- chi puo' approvare accesso source in team;
- quali bug evidence sono condivisibili tra device;
- quali framework vanno considerati P0 dopo Laravel.

## Final Note

La strada piu' breve verso l'obiettivo non e' aggiungere un altro sistema di
memoria. E' trasformare Hades backend in un evidence system: memoria condivisa,
graph aggiornato, source slices policy-compliant, bug evidence e freshness
devono lavorare insieme.

Quando questi P0 saranno completati, l'agent potra' spiegare una causa precisa
senza codebase live nei casi in cui il backend contiene evidence corrente e
copertura sufficiente. Nei casi mancanti, dovra' rispondere con precisione
diversa: non "non so", ma "mi manca X per distinguere causa A da causa B".
