# Production Readiness Plan - Hades Backend Kanban Agent Work - 2026-07-08

## Executive Summary

Questo piano copre la funzionalita' richiesta: i task scritti nel Kanban del
backend devono diventare lavoro eseguibile dagli agenti, con normalizzazione
intelligente, richieste di chiarimento quando il task e' ambiguo, uso della
memoria Hades condivisa e possibilita' per l'agent locale di scaricare, claimare
e gestire i task dal backend.

Stato attuale stimato per questa funzionalita': circa 35%. Le fondamenta
esistono, ma non sono ancora un flusso prodotto end-to-end:

- il backend ha gia' `agent_work_items`, lease, eventi, server agents,
  triage/clarifier e tool di memoria;
- il plugin API ha gia' endpoint per list/claim/heartbeat/complete/fail dei
  work item locali;
- Hades locale ha gia' `hades backend worker`, ma e' ancora un comando basso
  livello, fallisce in modo opaco se manca `backend.plugin_local_workspace_id`
  e non espone un flusso task-oriented;
- il Kanban dashboard non risulta ancora collegato automaticamente a bug
  intake, evidence, work item locale e memoria condivisa;
- i backend agents possono usare tool di memoria, ma serve un contratto
  obbligatorio e verificabile per leggere/scrivere nella stessa memoria Hades
  usata dall'agent locale.

Sequenza consigliata:

1. definire lo stato sorgente dei task e il contratto payload tra Kanban,
   Hades evidence, `agent_work_items` e worker locale;
2. completare setup locale plugin/workspace, cosi' `hades backend worker` non
   ritorna piu' `{"error": 1}` senza istruzioni;
3. automatizzare normalizzazione e chiarimenti lato backend;
4. creare il bridge Kanban -> Hades bug intake/evidence -> `agent_work_items`;
5. aggiungere comandi locali task-oriented e test end-to-end;
6. harden security, osservabilita', idempotenza, UX dashboard/desktop e
   runbook.

## User Answers And Assumptions

Confirmed facts from user answers:

- L'obiettivo e' poter usare l'agent da qualunque device mantenendo memoria.
- L'agent locale deve essere project-aware e, idealmente, capace di spiegare la
  causa precisa di un bug anche senza accesso diretto alla codebase durante la
  diagnosi.
- TencentDB Agent Memory e' escluso se non serve.
- I task Kanban backend devono poter essere normalizzati, chiariti, assegnati e
  gestiti da agenti backend e agent locale.
- Gli agenti backend devono usare la stessa memoria condivisa dell'agent locale.
- Il worker locale attuale e' stato provato e ha restituito `{"error": 1}`.

Confirmed facts from repository evidence:

- Hades locale ha gia' un worker plugin work item, ma non un flusso utente
  task-oriented.
- Il backend ha gia' `agent_work_items`, eventi, lease, plugin API e dashboard
  API per work item.
- Il backend ha gia' triage e clarifier agent, piu' tool di memoria/wiki.
- Il plugin API serve work item `assigned_agent_key = local_agent`.
- Il claim plugin richiede una `local_workspace_id` registrata per device e
  repository.
- Le route Hades bug reports/evidence, source slices, project awareness e memory
  search sono gia' presenti.

Assumptions inferred from evidence:

- Il Kanban dashboard usa la tabella `tasks` e non crea ancora automaticamente
  `agent_work_items` per `local_agent`.
- La normalizzazione dei task deve essere un servizio backend, non solo un
  prompt nell'agent locale, perche' il task nasce nel backend e deve poter
  chiedere chiarimenti all'autore prima del claim locale.
- Il worker locale deve restare opzionale e service-gated: nessun nuovo core
  model tool e nessuna crescita del prompt/tool schema principale.
- I backend agents devono scrivere facts solo con provenance/evidence, non
  direttamente come note libere prive di fonte.
- Desktop/frontend sono utili per UX e osservabilita', ma il percorso minimo
  deve funzionare da CLI.

Open decisions for the user:

- Quale agente backend deve essere il normalizzatore primario:
  `socrates`, `platon`, `aristoteles`, oppure un nuovo key esplicito come
  `hades_task_normalizer`.
- Se un task ambiguo deve bloccare in una colonna/stato `clarification_needed`
  oppure restare nel backlog con domande/commenti allegati.
- Se un task pronto deve essere assegnato automaticamente a `local_agent` o solo
  dopo una conferma manuale da dashboard.
- Quale modello usare per normalizzazione e chiarimento in produzione: modello
  piccolo economico per triage, modello piu' forte per bug/root cause, o routing
  per criticita'.
- Quanto source/evidence e' consentito salvare in backend per progetti sensibili
  prima di chiedere conferma.

## Area Plans

### Product And UX

Related tasks: P0-7, P2-1, P2-2.

Il flusso minimo deve essere CLI-first: setup, list, work, status e output
JSON spiegabile. La dashboard backend deve poi mostrare readiness e domande di
chiarimento. Desktop/web sono superfici di osservazione e controllo, non devono
duplicare la chat principale.

### Frontend

Related tasks: P2-1, P2-2.

Il frontend backend deve rendere visibile lo stato di un task: ambiguo, pronto,
in attesa local agent, claimed, failed, completed. Il frontend locale/desktop
deve mostrare setup mancante e task queue senza introdurre un secondo
transcript.

### Backend And API

Related tasks: P0-1, P0-3, P0-4, P0-5, P0-6, P1-2, P1-4.

Il backend e' il punto in cui normalizzare task, chiedere chiarimenti,
produrre evidence e creare work item. Le API plugin e Hades esistenti vanno
estese o orchestrate, non duplicate.

### Data

Related tasks: P0-1, P0-4, P0-5, P1-2, P1-5.

Le entita' chiave sono `tasks`, `agent_work_items`,
`agent_work_item_events`, `agent_work_item_leases`, project memory, bug report
ed evidence. La priorita' dati e' idempotenza: un task deve avere uno stato
leggibile e non generare duplicati.

### Security

Related tasks: P1-1, P1-5.

Il confine piu' delicato e' device/workspace authorization. Il worker locale
non deve diventare una scorciatoia per leggere task o source/evidence di
progetti non autorizzati. I segreti restano in `.env` o secret store, non in
`config.yaml`.

### Testing And QA

Related tasks: P0-8, P1-4, P2-4, P2-5.

Serve una matrice di test multilivello: unit/feature backend, test client CLI
con transport mock, contract test payload, smoke manuale con backend reale e
fixture no-source.

### Deploy And CI

Related tasks: P1-4, P1-6.

Il rollout richiede ordine: migrazioni/feature backend prima, poi update Hades
locale. La CI deve impedire che backend e agent divergano sullo schema payload.

### Observability

Related tasks: P1-3.

Ogni blocco deve produrre un codice stabile: setup missing, token missing,
workspace mismatch, project inactive, clarification needed, no queued items,
lease expired, awareness incomplete.

### Performance And Scalability

Related tasks: P2-3.

Il polling locale deve avere limit, backoff e concorrenza controllata. I task
non devono avviare agent run costosi se sono ancora ambigui o se manca
awareness minima.

### Reliability And Resilience

Related tasks: P1-2, P2-5.

Lease, heartbeat, retry e cross-device devono essere deterministicamente
testati. Un worker lento o morto non deve bloccare per sempre un task, ma un
worker con lease scaduto non deve completare lavoro non piu' suo.

### Accessibility

Related tasks: P2-1, P2-2.

Per la dashboard backend, task readiness e azioni devono essere accessibili da
tastiera, con errori testuali e focus gestito. Non e' P0 per il flusso CLI, ma
e' necessario prima di considerare completa la UX dashboard.
