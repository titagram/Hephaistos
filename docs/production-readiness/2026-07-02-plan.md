# Production Readiness Plan - Hades Agent - 2026-07-02

## Executive Summary

Hades Agent is a broad, multi-surface personal AI agent with a working local
core, CLI/TUI/desktop/dashboard surfaces, messaging gateway, plugins, skills,
cron, Docker packaging, and a new Laravel-backed Hades project integration.
The repository is not yet production-launchable as a backend MVP because the
core local backend path exists, but the launch path around it still needs to be
closed end to end: backend/dashboard token generation, one-line installer
onboarding, API contract verification, waiting-job/proposal review UX,
background sync, conservative data handling, and release/rebrand gates.

Recommended implementation sequence:

1. Close the Hades backend MVP launch path first: dashboard-generated bootstrap
   token and one-liner, installer bootstrap verification, project linking, and
   initial sync.
2. Lock the `/api/hades/v1` contract between this repo and the remote Laravel
   backend with fixture-backed and smoke-backed tests.
3. Make shared-memory, artifact, and job flows safe: redaction, retention,
   explicit review/approval, bounded payloads, and actionable degraded states.
4. Add lightweight background sync and production observability.
5. Only then finish public-release gates: full CI, packaging, Docker, desktop,
   docs, rebrand consistency, and support/runbooks.

Current readiness level: technical MVP foundation is present; production launch
requires P0 work before external users should depend on the backend flow.

- Evidence: `docs/mvp_checkpoint_2026-07-01.md` lists dashboard bootstrap token
  and installer/onboarding as P0; `docs/hades/installation.md` documents the

  - `tests/hermes_cli/test_hades_backend_cmd.py`
  - External Laravel backend/dashboard path to verify before implementation.
- What to change: Add or finish the backend/admin dashboard flow that creates a
  one-time project bootstrap token and emits POSIX/Windows install commands
  containing `--backend-url`, `--backend-project-id`,
  `--backend-project-token`, workspace, and project name. Keep local installer

### P0-4: Enforce Conservative Shared-Memory And Artifact Data Safety

- Area: Privacy/Compliance
- Objective: Prevent accidental leakage of personal data, absolute paths,
  secrets, or raw source through shared memory, job results, artifacts, doctor
  reports, and inbox events.
- Evidence: `SECURITY.md` states Hades is a personal single-tenant agent and
  only OS-level isolation is a boundary. `hades_backend_jobs.py` redacts file
  contents for `read_files`, but `sync_git_tree` walks workspace files and
  hashes bytes; docs require bounded artifacts and no raw source for AST.
  Retention/deletion/export decisions remain open in MVP docs.
- Affected paths:
  - `hermes_cli/hades_backend_jobs.py`
  - `hermes_cli/hades_backend_sync.py`
  - `hermes_cli/hades_backend_db.py`
  - `hermes_cli/hades_backend_client.py`
  - `hermes_cli/doctor.py`
  - `docs/hades/operations.md`
  - `docs/hades/backend.md`
  - `docs/hades/openapi-hades-v1.json`
  - `docs/LEGACY_COMPATIBILITY.md`
  - `tests/hermes_cli/test_hades_backend_sync_runner.py`
  - External Laravel backend storage/retention code.
- What to change: Define and enforce payload limits, redaction fields,
  retention windows, deletion/export behavior, and path/source disclosure rules
  for every backend-synced data class.
- How to change it: Add explicit schema fields for `truncated`, `redactions`,
  `omitted`, `retention_class`, and source provenance. Use streaming hashes for
  large files instead of `path.read_bytes()` in tree scans. Exclude ignored,
  generated, secrets, env, cache, and binary-heavy paths by policy. Require
  confirmation for raw content jobs or configure them as deny-by-default for
  public launch. Add docs and tests for deletion/export expectations.
- Dependencies: User decision on launch data posture; backend storage TTL and
  delete/export capability.
- Acceptance criteria:
  - No raw absolute local path is sent to the backend unless explicitly
    documented as a redacted display path.
  - Artifact uploads contain bounded metadata and omit raw source by default.
  - Doctor reports include aggregate counts only and never token, job payload,
    or absolute path contents.
  - Retention, deletion, and export policy is documented for memory proposals,
    accepted memory, jobs, artifacts, doctor reports, and inbox events.
- Verification:
  - `PYTHONDONTWRITEBYTECODE=1 .venv/bin/python -m pytest -q -p no:cacheprovider tests/hermes_cli/test_hades_backend_sync_runner.py tests/hermes_cli/test_hades_backend_doctor.py`

## Area Plans

### Product And UX

The primary launch journey should be treated as one product: backend dashboard
generates a tokenized command, local installer bootstraps Hades, workspace links
to backend, shared memory/jobs sync, and users see actionable degraded states
when anything pauses. Do not rely on manual SQLite edits, hidden dashboard DB
operations, or raw log interpretation for this path.

### Frontend

Related tasks: P0-5, P0-8, P2-1, P2-2.

The desktop and dashboard already have broad route surfaces, but backend MVP
status is currently compact and partially surfaced. Make CLI/TUI RPC payloads
canonical first, then add frontend affordances. Bootstrap installer typecheck
must be rerun after a clean dependency install.

### Backend And API

Related tasks: P0-2, P0-3, P0-6, P1-4.

The local client and OpenAPI fixture exist. The production risk is drift between
the local client, docs fixture, and remote Laravel backend. Every status,
proposal, job, artifact, doctor, and inbox shape needs contract tests.

### Data

Related tasks: P0-4, P1-1, P1-5.

Data classes include local SQLite state, shared memory cache/proposals, job
payloads/results, artifacts, doctor reports, and inbox events. Define retention,
deletion/export, redaction, truncation, and omission semantics before public
launch.

### Security

Related tasks: P0-4, P1-3, P2-4.

The existing trust model is clear: only OS isolation is a boundary. Production
readiness depends on not overclaiming in-process safety, keeping network
surfaces allowlisted/authenticated, and making data sent to the backend bounded
and explicit.

### Testing And QA

Related tasks: P0-8, P1-4, P2-2.

Safe checks show many local gates are healthy, and focused Hades tests pass.
Before release, rerun full CI on a clean branch/install, add Hades MVP smoke,
and validate frontend/build/package surfaces that analysis mode skipped.

### Deploy And CI

Related tasks: P0-1, P0-8, P1-6, P2-4.

CI is well developed, but Docker is intentionally not part of the required
aggregate gate today. Decide what is required for each release target and make
that explicit in docs/branch protection/release checklist.

### Observability

Related tasks: P1-2, P1-7.

Observer hooks exist for agent execution; backend sync and plugin work item
operations need similarly useful sanitized logs, counters, doctor report
payloads, and runbooks.

### Performance And Scalability

Related tasks: P1-5.

Large workspaces can stress file walking, hashing, AST extraction, and payload
upload. Add budgeted, streaming, gitignore-aware behavior before relying on
artifact sync in real monorepos.

### Reliability And Resilience

Related tasks: P0-6, P1-1, P1-2.

Manual sync works, but production use needs automatic/piggyback sync, backoff,
timeout, no duplicate concurrent sync, and clear recovery from backend outages.

### Accessibility

Related tasks: P2-2.

No full accessibility audit was run. Treat keyboard navigation, focus, error
visibility, and responsive behavior as a P2 product-readiness gate for public
UI polish, after P0 backend launch blockers are closed.

### Privacy And Compliance

Related tasks: P0-4, P1-1.

Privacy is a P0 blocker for backend launch because Hades handles personal
memory and local source/workspace context. Do not launch shared memory/artifact
sync without explicit retention, deletion/export, redaction, and payload limit
policy.

### Support And Operations

Related tasks: P1-2, P1-7, P2-4.

Support needs a safe doctor-report path, local recovery commands, backend
operator runbooks, and clear escalation. The docs have a foundation but should
be rewritten from the user symptom forward.

### Documentation

Related tasks: P0-2, P0-7, P1-7, P2-3.

The repo has strong internal docs. Production docs should move stable launch
instructions into `docs/hades` and website docs, with coordination logs treated
as historical/internal evidence.
