# Production Readiness Plan - Hades Agent - 2026-07-02

## Executive Summary

Hades is not ready for an unmanaged public production launch yet. It is much
closer than the previous readiness note suggested: the tracked worktree was
clean during the audit, the Hades backend client surface is implemented, docs
audit passed, lint passed, all targeted Python Hades tests passed, and the
TypeScript packages checked in this audit all typechecked.

The remaining gap is not "nothing works". The gap is that the original launch
contract asks for a coherent product that a user can install, connect to a real
backend, use from the primary surfaces, and trust for memory and delegated work.
Several of those pieces still rely on CLI-only fallback, staging assumptions,
stale documentation, or incomplete contracts.

Current readiness judgement:

- Controlled technical beta: plausible after the P0 scope and live backend gates
  below are closed.
- Public production release: not ready.

Top blockers:

1. Release scope is not frozen. Public launch, controlled beta, and internal
   dogfood have different requirements.
2. The installer and user-visible copy still expose legacy Hermes branding and
   non-final branch or URL assumptions.
3. The desktop and dashboard surfaces mostly display backend status; important
   job and memory-proposal actions are still CLI-first.
4. Shared-memory mutation semantics are incomplete for production use. Create
   proposals exist, but update/delete identity, versioning, and stale-state UX
   need to be explicit.
5. `project_inspection` is documented as an intelligent backend job but the
   local implementation currently returns a bounded git tree manifest.
6. Live remote Laravel backend proof was not run in this local read-only audit.

## Scope And Assumptions

The user asked whether the project is really ready and whether it respects the
initial requests. No additional answers were provided during this audit, so the
plan uses these assumptions:

- "Ready" means at least the first milestone in
  `docs/implementation_plan.md`: installable and launchable Hades, coherent
  branding/defaults, desktop main surface validated, technical CLI/TUI,
  dashboard metrics, real Laravel backend token/project connection, shared
  memory, Persephone contract operational or ready for E2E, curated defaults,
  and green docs/audit gates.
- The Laravel backend is considered part of the ready state, even though its
  source is not in this repository.
- The first acceptable release can be a controlled technical beta only if that
  is explicitly documented and the missing public-launch items are not marketed
  as finished.
- This audit is read-only for application code. Only this readiness plan was
  intentionally changed.

Open decisions before shipping:

- Is the first release public, private beta, or internal dogfood?
- Is the canonical install branch `main`, a version tag, or this
  `codex/hades-rebrand` branch?
- Which domain is canonical: `home-sweet-home.cloud`, local development hostnames,
  or another production URL?
- Which data classes may leave the local machine for the backend?
- Must desktop/dashboard be fully actionable on day one, or is CLI-first backend
  operation acceptable for beta?
- Is Persephone required as realtime push, or is polling acceptable for beta?
