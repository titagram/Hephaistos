# Production Readiness Plan - Hades Agent Project Awareness - 2026-07-06

## Executive Summary

Hades has a working backend-memory MVP, but it is not yet ready for the target
state requested in this analysis: login from any device with durable memory, and
project work where the agent has evidence-backed awareness of project structure,
history, decisions, source-derived facts, and freshness even when it does not
have direct filesystem access to the codebase.

The current implementation already has useful foundations:

- backend bootstrap, agent registration, workspace binding, shared-memory
  snapshots/proposals, jobs, artifacts, Persephone inbox polling, dashboard
  status/actions, and a Hades memory provider;
- per-turn external-memory prefetch is injected into every LLM API request made
  inside the same turn;
- read-only project jobs can upload git tree metadata and Python symbols;
- the backend, from prior same-thread SSH inspection, already exposes dashboard
  project memory/wiki/graph surfaces and server-side search tools.

The main blockers are not storage-engine problems. TencentDB Agent Memory is
explicitly out of scope for this plan. The blockers are product and contract
shape:

1. The local Hades provider currently recalls the first cached memory summaries,
   not query-ranked project context.
2. Raw/wiki file chunks can land in generic memory and pollute automatic recall.
3. The local agent has no service-gated tools to search backend memory, wiki,
   graph, or exact source chunks on demand.
4. A workspace without binding silently loses project memory instead of entering
   a guided mapping/bootstrap flow.
5. Source-derived knowledge is still metadata-tree plus Python-symbol MVP, not a
   complete code-intelligence substrate.
6. Freshness is not yet a hard contract tied to commit/index state.
7. Cross-device personal memory is not separated from project shared memory as a
   first-class backend domain.

Recommended implementation sequence:

1. Make the backend Hades project brain the only source of truth.
2. Separate memory domains before improving recall.
3. Add query-aware recall/search APIs and local service-gated tools.
4. Fix the local provider so automatic injection is compact, filtered,
   freshness-aware, and tested for every model API iteration.
5. Build the project intelligence layer: wiki, logbook, agent notes, graph,
   source chunk index, schema/routes/dependencies, and commit freshness.
6. Add UI, doctor, E2E, and operational gates that prove awareness is current
   and evidence-backed.

Implementation update, 2026-07-06:

- Completed the first agent-side slice in this repository:
  - `hades_backend.prefetch()` now ranks cached backend memory by the current
    query instead of injecting the first cached summaries.
  - Automatic recall excludes raw source/wiki chunk entries such as
    `hades.backend_wiki.file_chunk.v1`.
  - The provider exposes the service-gated
    `hades_backend_project_memory_search` tool for explicit cached project
    memory search, including domain filters and opt-in raw chunk retrieval.
  - Unmapped workspaces now surface a clear no-binding state in the provider
    prompt block and search tool response.
  - A focused agent-loop regression proves prefetched memory is injected into
    every API request in a multi-iteration tool turn while persisted messages
    remain clean.
- This does not complete the backend/API side of complete project awareness.
  P0-1, P0-3, freshness contracts, graph/source indexing, dashboard controls,
  and no-codebase E2E proof remain required.

  - The core model toolset should remain narrow; Hades-aware search/fetch tools

## Roadmap

### P0 - Blocking Foundations

### P0-1: Define The Hades Project Brain Contract

- Area: Backend/API
- Objective: Create a single Hades-owned contract for complete project
  awareness that separates memory, wiki, source index, graph, artifacts,
  freshness, and user identity before changing recall behavior.
- Evidence: `docs/hades/backend.md` states the backend owns shared memory;
  `docs/backend-agent-coordination.md` already distinguishes memory domains and
  project artifacts; `docs/hades/openapi-hades-v1.json` currently documents the
  backend MVP but not query-aware project brain retrieval.
- Affected paths:
  - `docs/hades/backend.md`
  - `docs/hades/openapi-hades-v1.json`
  - `docs/hades/operations.md`
  - `docs/backend-agent-coordination.md`
  - `docs/implementation_plan.md`
  - Remote Laravel backend, likely `app/Http/Controllers/Hades/*`,
    `app/Services/*`, and database migrations; verify exact paths before
    editing backend code.
- What to change: Define versioned project-brain domains and contracts:
  `personal_memory`, `project_memory`, `logbook`, `agent_notes`, `wiki`,
  `source_chunks`, `code_graph`, `artifacts`, `freshness`, and
  `retrieval_audit`.
- How to change it: Add a contract document and OpenAPI expansion that states
  which domain owns each data class, which data can be auto-injected, which
  data is only retrievable on demand, and which writes are proposals versus
  authoritative backend mutations.
- Dependencies: User decisions on source-content retention and personal/team
  sharing policy.
- Acceptance criteria:
  - The contract says that Hades backend is the only shared-memory authority.
  - The contract forbids raw source/wiki chunks from generic automatic memory
    injection.
  - The contract defines freshness fields such as `indexed_commit`,
    `workspace_head_commit`, `index_status`, and `stale_reason`.
  - The contract names the tool/read APIs the local agent is allowed to use.
- Verification:
  - `python3 scripts/docs_audit.py`
  - `rg -n "project brain|personal_memory|project_memory|source_chunks|indexed_commit|stale_reason" docs/hades docs/backend-agent-coordination.md docs/implementation_plan.md`

### P0-2: Separate Memory Domains And Quarantine Raw Chunks

- Area: Data
- Objective: Stop exact file/wiki chunks from polluting automatic shared-memory
  recall while preserving them as retrievable, provenance-rich backend
  knowledge.
- Evidence: `plugins/memory/hades_backend/__init__.py` currently injects cached
  summaries without schema filtering; the user-provided example uses
  `schema=hades.backend_wiki.file_chunk.v1`; `skills/autonomous-ai-agents/backend-knowledge-sync/SKILL.md`
  treats chunking as exact archive/import behavior.
- Affected paths:
  - `plugins/memory/hades_backend/__init__.py`
  - `hermes_cli/hades_backend_cmd.py`
  - `hermes_cli/hades_backend_db.py`
  - `hermes_cli/hades_backend_client.py`
  - `tests/agent/test_hades_backend_memory_provider.py`
  - `tests/hermes_cli/test_hades_backend_sync_runner.py`
  - Remote Laravel backend memory/import services and migrations; likely
    `MemoryImportService`, memory proposals, project memory entries, wiki
    pages/revisions, and artifact tables.
- What to change: Add domain/schema metadata handling so automatic recall
  includes only concise `project_memory`, `logbook`, and `agent_notes` entries.
  Move or tag `hades.backend_wiki.file_chunk.v1` as source/wiki chunk content
  excluded from automatic memory.
- How to change it: Implement local filtering in the provider first, then add
  backend migration/import policy so future import bundles write chunks to wiki
  or source chunk storage instead of project-memory summaries. Provide a cleanup
  command or backend migration for existing chunk-like memory entries.
- Dependencies: P0-1 domain contract.
- Acceptance criteria:
  - Provider prefetch ignores cache items with chunk/source schemas by default.
  - Existing chunk batches remain reconstructable through source/wiki retrieval.
  - New memory import bundles cannot silently create hundreds of auto-recall
    chunk summaries.
  - Dashboard/backend status exposes how many entries were quarantined or
    remapped.
- Verification:

  `hermes_cli/hades_backend_cmd.py` syncs the full snapshot into local cache

### Product And UX

Related tasks: P0-6, P1-3, P1-4, P2-5.

The product must make project binding, freshness, and knowledge coverage
visible. Silent no-memory operation is the main UX risk. Dashboard can be the

### Frontend

Dashboard already has a backend page with jobs and proposal actions. Extend it

### Backend And API

Related tasks: P0-1, P0-3, P0-4, P0-5, P1-1, P2-2.

The backend API must evolve from snapshot/proposals/jobs into a project-brain
retrieval surface. Keep local Hades as client/cache/outbox. Do not move
authoritative project memory into frontend or local SQLite.

### Data

Related tasks: P0-2, P0-7, P1-1, P2-4.

The highest-risk data issue is mixing raw chunks with memory summaries. Separate

Related tasks: P0-5, P0-6, P1-6, P2-3.

Provider tools must be service-gated by backend config and workspace binding.
Tool calls must not cross project/workspace boundaries. Logs and support output

### Testing And QA

Related tasks: P0-4, P0-5, P1-5, P1-6.

Current targeted tests are green. Missing coverage is the no-codebase E2E,

### Deploy And CI

Related tasks: P1-5, P2-5.

No deploy change is required to start local implementation. Before release,

### Observability

Awareness state must be visible in `hades backend status --json`, `hades

### Performance And Scalability

Personal memory, project memory, graph metadata, and source chunks have

Related tasks: P1-6, P2-5.

### Documentation

Related tasks: P0-1, P1-2, P2-5.

Docs must distinguish exact archive chunks from memory, automatic recall from

## Risks And Decisions

- "Complete project awareness" is impossible if the backend index is stale,
  truncated, policy-blocked, or missing language/framework extractors. Hades
  must surface that state rather than claim certainty.
- Exact source chunks are powerful but sensitive. The user must choose whether
  source content is backend-stored, fetched on demand after local confirmation,
  or disallowed for some projects.
- If automatic recall keeps raw chunks, prompt quality will degrade and costs
  will rise. P0-2 should happen before large imports continue.
- Adding Hades search tools as core tools would violate the narrow-waist
  guidance. Keep them provider/plugin gated.
- Per-turn recall is currently computed once and reused on every API call in
  the turn. If future behavior requires mid-turn refresh after a backend sync
  tool call, that should be a separate design because it can affect prompt-cache
  assumptions.
- Cross-device memory needs identity and privacy policy. Project binding alone
  is not enough.
- Backend and local repo must evolve together. The OpenAPI fixture in this repo
  should stay close to the Laravel implementation and live staging smoke should
  catch drift.
